## 2026-05-18 - paths.py (config layering)
- Bumped `requires-python` from `>=3.10` to `>=3.11` so we can hard-require stdlib `tomllib` and skip the `tomli` backport. Dropped the 3.10 classifier in pyproject.toml. Rationale: pure-stdlib TOML parsing is simpler than an optional dep and the user's environments are all 3.11+.
- Added `[tool.pytest.ini_options]` with `pythonpath = ["src"]` so `python3 -m pytest` works without an editable install (PEP 668 blocks `pip install -e .` on this Ubuntu system Python).
- Enforced the 12k char ceiling explicitly: `hard_limit` must be strictly less than `HARD_LIMIT_CEILING = 12000`. The constant lives at the top of `paths.py` so other modules can import it if they need the same bound. Reasoning: hard_limit is supposed to leave headroom before content truncation kicks in around 12k, so allowing `hard_limit = 12000` defeats the safety margin.
- `cache_dir` is auto-created at config-resolution time (it's a cache, that's fine). `workspace_dir` and `cards_dir` are never auto-created; `allow_missing_cards=True` only relaxes `cards_dir` (and only if the parent dir exists, so we can create it later in `trim`).
- Default config-file lookup at `~/.config/bootstrap-doctor/config.toml` is best-effort (missing is fine). An explicit `--config` flag or `BOOTSTRAP_DOCTOR_CONFIG` env var that points at a missing file raises `ConfigError`, because that's almost always a typo.
- Env-var coverage for v1: only the six knobs in the spec (`WORKSPACE_DIR`, `CARDS_DIR`, `GATEWAY_URL`, `GATEWAY_MODEL`, `SOFT_LIMIT`, `HARD_LIMIT`). `tracked_files`, `named_workspaces`, heuristics, and cache dir stay config-file-only; they're list-shaped or namespaced and don't map cleanly to env strings.

## 2026-05-18 - parsing.py
- CRLF normalization happens once at the top of `parse_text` via a literal `\r\n` -> `\n` replace (plus lone `\r` -> `\n` as a defensive second pass). All downstream measurements (`char_count`, `line_count`, `start_line`, `end_line`) run on the LF-normalized form, so two files that differ only in line-ending convention produce identical Section output. `parse_file` reads bytes (not text) so Python's `open()` newline rewriting doesn't pre-empt this and silently change what we measure. Char counts are therefore the LF-normalized count, which is what the heuristics threshold (400 chars) and the 10k/11.5k limits should compare against in a platform-stable way.
- H4 and deeper headings (`#### `, `##### `, ...) intentionally do NOT split sections; they stay in the parent section's body verbatim. Implementation gotcha: the H3 regex `^###\s+` also matches `#### Foo` (since `### ` is a prefix of `#### `), so we explicitly guard with a separate `H4_PLUS_RE` check before treating a line as an H3. Without that guard, an H4 would mis-split as an H3 named `# Foo`. Captured in the `test_four_pound_heading_not_split_as_h3` test.
- Code-fence-aware splitting: maintain a single `in_fence` bool that flips whenever we see a line whose stripped form starts with ``` (three backticks). The fence flag is consulted before every heading check, so a `## fake heading` inside a fenced block is appended to the parent section's body and never starts a new section. Fences in the preamble are tracked too, so a preamble that opens a fence containing `## Real` doesn't get cut short mid-fence.
- Preamble emission rule: a preamble Section is only emitted if there's at least one non-blank line before the first H2/H3. Leading blank lines alone do not manufacture an empty preamble. This keeps `parse_text("\n\n## Title\nbody\n")` from emitting a noisy zero-body preamble that would clutter every heuristics run.
- Heading text normalization strips outer whitespace and a trailing run of `#` (with optional inner whitespace), per the ATX closure idiom (`## Title ##` -> `Title`). Inline markdown is preserved as-is, so `Tools (local)` and `**Bold**` survive verbatim. This matches the spec's "do NOT process inline markdown" requirement.
- `last_touched_git_mtime` walks parent dirs looking for a `.git` entry (file or dir, to cope with worktrees and submodules), then runs `git log -1 --format=%ct -- <relative_path>` from the repo root. Empty stdout (untracked file), non-zero exit, subprocess failure, and a missing input file all yield `None` without raising. Subprocess has a 10s timeout to guarantee the function returns even if git hangs.

## 2026-05-18 - safety.py
- `atomic_write_text` puts the tempfile in the same directory as the target via `tempfile.mkstemp(dir=path.parent, ...)`. Same-directory placement is the only way `os.replace` is guaranteed atomic on POSIX, because rename across filesystem boundaries falls back to copy+unlink. Parent dirs are auto-created (`mkdir parents=True, exist_ok=True`) so callers can write the first card without a pre-step. On any exception we best-effort `os.unlink(tmp)` and re-raise; swallowing OSError on the cleanup so a missing-tempfile race never masks the original failure.
- `_find_repo_root` walks up from the supplied dir looking for a `.git` entry (file or dir, mirrors the same trick `parsing.last_touched_git_mtime` uses). This means `assert_git_clean` works from any subdir inside the workspace, not just the worktree root. If the walk hits the filesystem root without finding `.git`, we raise DirtyWorkspaceError("not a git repo: ...") rather than silently treating it as clean - "no repo" should refuse to mutate, since there's nothing to revert against.
- `assert_git_clean` treats subprocess timeout, OSError, and non-zero exit as DirtyWorkspaceError, not as "clean enough to proceed". The spec is explicit that a probe failure must not turn into a green light. `--porcelain` output is parsed line-by-line; with `allow_untracked=True` we drop lines starting with `??` and re-check, so the user can opt into "I just created some cards, that's fine."
- `slugify` boundary-preferring truncation: after producing the lowercase-dashed form, if the result is longer than 80 chars we look for the last dash inside `[70, 80)` and cut there. No dash in the window means a hard cut at 80. After cutting we `strip("-")` again so we never emit a trailing dash if the truncation point happened to land right next to one (covered by `test_slugify_strips_trailing_dash_after_truncation`).
- `resolve_card_target` rejects on a deny-list of structural traps (empty, whitespace-padded, `/`, `\`, `\n`, `\0`, `\r`, `.`, `..`, leading `.`) before composing the path, then runs the resolved candidate through `ensure_within` for defense in depth. The leading-dot ban exists because hidden cards would be invisible in `ls memory/cards/` and would never surface in audit output - a card the operator can't see is a class of failure we'd rather refuse outright.
- `ensure_within` uses `Path.is_relative_to` (Python 3.9+, fine under the 3.11+ floor). Both base and candidate are `.resolve()`d once, so symlinks pointing outside the cards dir are caught even when the lexical path looks clean.

## 2026-05-18 - status.py
- Severity precedence is explicit: `missing` > `unreadable` > `hard` > `soft` > `ok`. A missing file short-circuits before we ever try to stat or read it, so file-system races (file appears between `exists()` and `stat()`) downgrade gracefully to `unreadable` rather than crashing. `unreadable` is reserved for files that exist but fail `read_text(encoding="utf-8")` (decode error or OSError). The two failure modes are deliberately distinct severities so an operator can tell "I forgot to create the file" from "this file has invalid bytes."
- Exit code mapping: `_exit_code` scans the row list once. The first `hard`/`missing`/`unreadable` short-circuits to `2`; otherwise we keep scanning and return `1` if any `soft` was seen, else `0`. `2` dominates `1` dominates `0`, matching unix convention where higher codes signal more severe failures.
- `_count_lines` counts `\n` and adds 1 if the last char is not `\n` (so `"a\nb"` is 2 lines, `"a\nb\n"` is also 2 lines). Empty string is 0 lines. Matches `wc -l` for files ending in `\n` and is one greater than `wc -l` for files without a trailing newline (which is the more useful count for a human staring at a buffer).
- Limit-boundary semantics match the spec: `chars >= hard_limit` is `hard`; `soft_limit <= chars < hard_limit` is `soft`; `chars < soft_limit` is `ok`. So a file at exactly `soft_limit` chars is `soft`, and one at exactly `hard_limit` is `hard`. Easier to reason about than `>` boundaries and produces fewer "off by one" complaints when the operator manually pads a file to find the cutover.
- Workspace iteration order is `(primary, *named_workspaces)`, with the primary labeled `"workspace"` and each named entry labeled by its own dir name. `_workspace_scopes` runs every named entry through `safety.ensure_within(workspace_dir, workspace_dir / name)` defensively. An escape would be a config validation miss (the validator already rejects `/` in entries), but propagating `UnsafeTargetError` here means a future config-loader regression cannot silently scan outside the workspace tree.
- Missing named workspace dirs do NOT crash the run. `collect()` still emits one `missing` FileStatus per tracked file in that workspace, and `render_text()` prepends a `warning: named workspace 'X' does not exist: ...` line so the operator can disambiguate "I deleted that dir" from "all my files vanished." The JSON renderer skips the warning entirely - missing rows are self-describing in JSON form.
- Stdlib-only by design. The text renderer hand-builds a width-aligned table with `format()` minilanguage rather than pulling in `rich`/`tabulate`. The judgment call: this CLI ships as a pipx-installable diagnostic tool with one already-justified runtime dep (`requests`, for `judge.py`); adding `rich` doubles the install surface for a verb that prints fewer than 50 lines of output. If colorization becomes a real ask, gate it behind `sys.stdout.isatty()` and emit ANSI by hand.
- `render_json` emits `path` as a string (not a Path object, which is unserialisable). Top-level `soft_limit`/`hard_limit` are duplicated outside `rows[]` so a downstream consumer doesn't have to re-derive the limits to colorize their own output. JSON keys match FileStatus field names verbatim so a consumer can `FileStatus(**row)` round-trip if they import the dataclass.
- Test helper (`make_cfg`) writes a per-test `config.toml` rather than constructing `Config` directly, because `tracked_files` and `named_workspaces` are config-file-only in v1 (no env-var or kwarg path). This keeps the test harness honest about what the production loader can express, and exercises `resolve_config` as a side effect on every test.

## 2026-05-18 - heuristics.py
- Deterministic mtime injection: `is_stale` takes `last_touched_ts` as a kwarg instead of computing it itself, and `shortlist` (the only place that touches the filesystem) accepts `now_ts` for the same reason. This keeps every heuristic pure: every test injects a fixed epoch (`now = 1_700_000_000`) plus a synthetic file mtime and gets the same result on every machine. The `time.time()` fallback inside `is_stale` is the only non-deterministic path and is exercised by exactly one test that monkeypatches `time.time`. Without this discipline, "is_stale" tests would flake whenever the test machine clock advanced past a day boundary mid-run.
- Long-code-block: a SINGLE block must clear the line threshold; two short blocks do NOT aggregate. Reasoning: the heuristic is meant to flag content like "## Cron Setup" sections that are 95% a giant config dump, not sections that happen to have a couple of three-line shell snippets. Aggregating would fire on any section with a bunch of small inline examples (cookbook-style content), which is exactly the kind of section we want to KEEP inline. Implementation walks lines once with an `in_fence` flag, counting non-fence lines while in_fence and resetting on close. Unclosed fences (open ``` with no matching close) are intentionally not honored - the section already passed the parser, so an unclosed fence here means the body was clipped mid-block and counting "everything to end of body" would be misleading.
- Duplicate threshold pinned at 0.80 / 100 chars for v1: tuned by hand against real bootstrap content. Below 0.80 we picked up coincidental phrasing matches between two unrelated "## Recent Session" sections sharing stock prose ("Filed PR, all green, restart gateway"). Above 0.85 we missed paraphrased copies that a human would clearly call "the same thing in two files". 0.80 is the balance point. The 100-char floor exists because `difflib.SequenceMatcher.ratio()` is noisy on tiny bodies - "see above" and "TODO" hit 0.6+ against random other short bodies. These are kept as module-level constants, NOT config knobs, because adding a knob invites users to "tune" the heuristic without a way to compare runs. If the threshold is wrong we want to fix it once in code, not split the user base into N different shortlist behaviors.
- Normalization for duplicate detection: lowercase + whitespace-run-to-single-space + outer-strip, applied once per section before any pairwise comparison. We intentionally do NOT strip punctuation or markdown formatting - two sections that differ only in `**bold**` vs plain text are still meaningfully different ("did someone reformat this?"), and the SequenceMatcher ratio degrades gracefully on those.
- Preamble exclusion (heading_level == 0): preamble sections are the text before the first `## ` heading. They're typically file-level intros ("This file holds X. See spec for details.") that don't move cleanly to a card - cards have their own front matter and topic. Excluded from BOTH the per-section reason pass AND from duplicate detection in either direction, so a "duplicate preamble across two files" never surfaces. If a preamble is genuinely too large, that's a manual fix, not a card-promotion candidate.
- shortlist mtime caching: `last_touched_git_mtime` shells out to `git log`, which is 50-200ms on a warm cache. With 8 tracked files and ~30 sections per file, doing this per-section would burn 6-12 seconds per `bootstrap-doctor` run for no benefit (every section in one file has the same git mtime). Cache is a simple `dict[Path, int | None]` keyed by `section.file`; we populate it lazily with a "first section in this file" check. Test `test_shortlist_mtime_cached_per_file` verifies one call per unique file by patching `last_touched_git_mtime` with a recorder.
- Reasons list ordering: deterministic within a section as `["large", "long-code-block", "stale", "duplicate"]`. The judge prompt eventually formats this as "reasons: large, stale" and we want a stable order so prompt-cache hashes stay stable across runs (otherwise every change to a section's body could re-order its reasons and bust the verdict cache).

## 2026-05-18 - judge.py
- Prompt design: system prompt is a verbatim module constant (`SYSTEM_PROMPT`) so prompt-cache hashes stay stable across runs and across CI invocations. It enumerates the full taxonomy and the strict JSON schema inline, which is redundant with the validator but gives the model a chance to comply on the first try - the verbose schema chatter is cheap compared to a retry. Temperature pinned to 0.0 and `response_format={"type": "json_object"}` is set even though not every gateway honors it (OpenClaw's GPT 5.5 path does); the validator is the real safety net. User prompt is small and stable: file name + heading path + char count + reasons + raw body inside `<section body>...</section body>` tags. We do NOT pre-truncate the body; if a section is so large that the prompt blows past `max_tokens=400` on the response, that's a "move" candidate by sheer size and the model can still emit a tiny JSON verdict.
- Cache schema version=1: bumped only when the cache format itself changes incompatibly (new required fields, renamed keys). Verdict CONTENT changes (e.g., a tweak to the system prompt that changes which sections get "move") do NOT bump the schema version - they're handled by the SHA256 body key. If the body changes, the cache miss re-asks the gateway; if only the prompt changes but bodies are stable, the operator can just delete `verdicts.json` to force a re-judge. Schema mismatch reads the file, sees `version != 1`, prints a stderr warning, and starts fresh (the cache rebuilds on the next save). We don't try to "migrate" old caches: a verdict is cheap to regenerate compared to the engineering cost of N-version migration code.
- Failures aren't cached: a `judge_error: <err>` verdict means EITHER the gateway hiccupped (HTTP 500, network blip), or the model emitted garbage we couldn't parse. Both states are transient by definition - the next run might land on a healthier gateway, or the model might emit valid JSON on a retry. Caching the error would pin the candidate to "unsure forever" and the operator would never get a real verdict without manually clearing the cache. `budget_exceeded` follows the same logic: a future run with a higher `max_input_chars` (or fewer candidates ahead of it) might fit. So `_is_uncacheable()` is `decision=="unsure" AND reasoning starts with "judge_error:" or equals "budget_exceeded"`. Plain "unsure" verdicts from the model ARE cached - those are legitimate "I genuinely can't tell" judgments that won't change on retry.
- Budget-exceeded simple-bail strategy: the spec offered two options ("skip and continue to give later candidates a chance" vs "bail entirely on first overrun"). Picked the simpler "bail entirely" because (a) the test contract is easier to write and verify, (b) once we've crossed the budget on a given run, every remaining candidate also pays the system-prompt fixed cost on TOP of its own user prompt, so giving them "a chance" rarely fits more in practice, (c) an operator who hit the cap can re-run with a higher `max_input_chars` and pick up the un-judged candidates from cache + a fresh round on the rest. Implementation flips a single `budget_blown` bool and short-circuits every remaining iteration. `budget_exceeded` verdicts are NOT cached (see above).
- `requests_made` counts SUCCESSFUL gateway calls only; failures bump `failures` instead. The invariant `requests_made + cache_hits + failures == len(candidates)` only holds when no budget bail occurred. When the budget is blown, the budget_exceeded verdicts aren't counted in any of the three buckets - they're implicitly the remainder. This keeps `requests_made` meaningful as "how much did we spend on the gateway this run" rather than mixing in error counts.
- `http_post` injection seam pattern: the kwarg accepts `Callable[[str, dict], Any]` where `Any` must duck-type as a `requests.Response` (just `.status_code` and `.json()`, plus `.text` for the error-log path). Tests pass a tiny `FakeResponse` class with those three attributes and never touch the real `requests` module - confirmed by a `monkeypatch`-based test (`test_default_http_post_uses_requests_post`) that exercises the default path with `requests.post` itself swapped out. The seam is a kwarg rather than a class because there's only one HTTP call site and zero state between calls; a `GatewayClient` class would be ceremony for nothing.
- `_extract_content` is defensive about the chat-completion envelope: model providers occasionally return `choices: []` (refusal) or `message: null` (content filter), and we surface those as `JudgeError("response missing 'choices' array")` / `JudgeError("first choice missing 'message'")`. Both end up in the standard `judge_error:` path; the operator sees the actual provider quirk in the stderr log alongside the first 500 chars of response body, so debugging a flaky gateway doesn't require a packet capture.
- Code-fence-stripped JSON parsing: even with `response_format={"type": "json_object"}` set, some gateway backends (or older models routed through OpenClaw's auto-fallback chain) wrap their output in a ` ```json ... ``` ` fence. `_parse_verdict_json` strips one layer of fence before calling `json.loads`, so we tolerate that quirk without retrying. We don't try to fish JSON out of free-form prose - if the content isn't either bare JSON or fenced JSON, that's a `judge_error:` and a chance to escalate the gateway.
- Category coercion (not failure): if `decision=="move"` and `category` is outside the fixed taxonomy, we coerce to `""` rather than raising `JudgeError`. The model is otherwise behaving (decision + topic + hook are all valid), so dropping a slightly hallucinated category is friendlier than throwing away the whole verdict. Topic and hook on `move` ARE required to be non-empty strings - those drive the card filename and the breadcrumb the operator sees in the audit table, so accepting empties would silently produce unusable cards.
- `BOOTSTRAP_DOCTOR_GATEWAY_TOKEN` env var becomes `Authorization: Bearer ...`. Resolved at request time (not at config-resolution time) so an operator who exports the token mid-session doesn't need to restart anything. For localhost OpenClaw the token is usually unset and we send no auth header at all, which the gateway accepts on the loopback interface.

## 2026-05-18 - trim.py
- Card frontmatter shape reality-matched against `~/.openclaw/workspace/memory/cards/active-ports.md` and `agent-context-stamper-profiles-and-hygiene-2026-04-22.md`: keys are `topic`, `category`, `tags`, `created`, `updated`. Tags use inline-list syntax `tags: [a, b, c]` (no quotes around items, comma-space separator). Real cards in the wild sometimes drop `created` (one of the two examples did), but our spec requires it, so we always emit both `created` and `updated` set to `today_iso`. Added two provenance lines (`source: bootstrap-doctor` and `source_file: ...`, `source_heading: ...`) that aren't in the existing convention; rationale is "where did this card come from" auditability without bloating the frontmatter humans actually scan. If this turns out to be noise we can drop the source lines without touching the trim flow itself.
- Reverse-line-order rewrite: when a single bootstrap file has multiple `move` actions, we sort them by `start_line` descending before rewriting. Editing the later section first leaves all earlier sections' line numbers intact, so we never recompute offsets and never re-parse mid-loop. This matters because `_replace_section_body` indexes by `section.start_line` directly; if we rewrote the first section first, every later section's `start_line` would now point at the wrong line in the modified text. The alternative would be re-parsing after each edit, which is correct but slower and (more importantly) opens a window for "second action no longer finds its heading" false skips. Reverse-order is a clean invariant.
- Collision policy default = `skip`: the safest default for a destructive verb. `overwrite` is opt-in (operator explicitly wants to replace an existing card, e.g. after a manual revert), and `rename` adds `-2`, `-3`, ... up to `-10` then gives up. The `-10` cap is arbitrary but the rename path is mostly a fallback for accidental slug collisions, not bulk loading; ten attempts is plenty. Spec said "max 10 attempts; otherwise skip" so we cap and skip rather than blow past.
- Idempotency via section re-parse at apply time: `apply_plan` re-reads + re-parses the bootstrap file fresh and looks the target section up by `heading_path + start_line` before rewriting. Match preference: exact `heading_path + start_line`, then unique `heading_path` fallback (handles small upstream edits that shifted line numbers without removing the section). If neither matches, the action skips with no card written. A second invocation of build_plan after a successful apply produces actions that all skip with "card already exists" because the cards directory now has the slugged file - the breadcrumb-only sections wouldn't even reach the judge in a full end-to-end re-run (they're under the heuristic size threshold), so the idempotency story is "card-collision-skip in the plan + size-filter-skip in the heuristics" - both layers refuse to re-do the work.
- Source-file-changed handling: if `bootstrap_path.exists()` is False at apply time, every action targeting that file is skipped with no error (the file was renamed, deleted, or moved out from under us; we'd rather skip than create an orphan card pointing at nothing). Same with read failures - any OSError on `read_text` skips the file entirely. The "section heading no longer matches" case is handled by `_replace_section_body` returning None when the line at `start_line - 1` doesn't start with `## ` or `### `; this catches "operator edited the heading text" too, not just deletions. Counted in `runtime_skipped` and added to the summary's `skipped` field.
- Diff preview (`render_plan`): unified_diff with `n=3` context lines per affected file, prefixed by `--- a/<rel>` / `+++ b/<rel>` paths relative to `cfg.workspace_dir`. NEW CARD blocks render before the diff section so an operator scanning top-down sees "what's about to be created" first and "what's about to be modified" second. Body preview is truncated to 80 chars with `...` suffix so a 5KB section doesn't blow up the audit output. The footer line uses literal "actions planned" + "would write N card(s)" + "would modify N bootstrap file(s)" phrasing so it reads naturally in dry-run mode (when `apply=False`).
- No em dashes anywhere in user-facing output (breadcrumb separator, render_plan headings, summary footer). The breadcrumb separator is a regular hyphen wrapped in spaces (`- See [topic](path) - hook`). Matches global writing conventions and stays parser-friendly for any downstream tooling that splits on " - ".
